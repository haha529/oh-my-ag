# Агенты

Агенты в oh-my-agent — это не обычные чат-боты. Каждый смоделирован по конкретной инженерной роли с определённой областью, инструментами и стандартами качества.

## Категории Агентов

Думайте о них как о вашей инженерной организации:

| Категория | Агенты | Чем Занимаются |
|-----------|--------|---------------|
| **Генерация идей** | oma-brainstorm | Исследование идей перед написанием кода |
| **Планирование** | oma-pm | Требования, декомпозиция задач, контракты API |
| **Реализация** | oma-frontend, oma-backend, oma-mobile, oma-db | Собственно кодирование |
| **Дизайн** | oma-design | Дизайн-системы, токены, паттерны UI/UX |
| **Инфраструктура** | oma-tf-infra | Облачное развёртывание с Terraform |
| **DevOps** | oma-dev-workflow | CI/CD, миграции, релизы |
| **Качество** | oma-qa, oma-debug | Ревью, аудит безопасности, исправление багов |
| **Локализация** | oma-translator | Естественный многоязычный перевод |
| **Координация** | oma-orchestrator, oma-coordination | Запуск и синхронизация агентов |
| **Git** | oma-commit | Чистые conventional commits |

## Как Агенты Выполняют Работу

Каждый агент следует одной дисциплине:

### 1. Charter Preflight

Перед написанием кода агенты выводят `CHARTER_CHECK`, объявляя:
- В какой области они работают
- Какие ограничения действуют
- Какие допущения они делают
- Как выглядит «готово»

Это позволяет рано обнаружить расползание скоупа и недопонимания.

### 2. Двухуровневая Загрузка

Агенты эффективны по токенам по замыслу:
- **Уровень 1** (`SKILL.md`, ~800 байт): Идентичность и правила маршрутизации — загружается всегда
- **Уровень 2** (`resources/`): Протоколы выполнения, сценарии ошибок, шаблоны кода — загружаются только при необходимости

Это экономит ~75% токенов по сравнению с загрузкой всего сразу.

### 3. Выполнение в Рамках Скоупа

Фронтенд-агент не трогает бэкенд-код. DB-агент не изменяет UI-компоненты. Каждый агент остаётся в своей зоне.

### 4. Контрольная Точка Качества

У каждого агента есть чек-лист для своей области. Работа не считается «готовой», пока чек-лист не пройден.

## Стратегия Workspace

Для мульти-агентных проектов отдельные workspace уменьшают конфликты при слиянии:

```text
./apps/api      → workspace бэкенд-агента
./apps/web      → workspace фронтенд-агента
./apps/mobile   → workspace мобильного агента
```

## Процесс Оркестрации

Когда вы запускаете мульти-агентный workflow:

1. **PM-агент** декомпозирует задачу на подзадачи по доменам
2. **Доменные агенты** выполняют параллельно, каждый в своём workspace
3. **Прогресс** передаётся в память Serena (`.serena/memories/`)
4. **QA-агент** проверяет кросс-доменную согласованность
5. **Результаты** собираются и готовы к слиянию

## Состояние Времени Выполнения (Память Serena)

Агенты координируются через общие файлы памяти:

| Файл | Назначение |
|------|-----------|
| `orchestrator-session.md` | Состояние текущей сессии |
| `task-board.md` | Назначения задач и статус |
| `progress-{agent}.md` | Обновления прогресса по агенту |
| `result-{agent}.md` | Финальные выходные данные агентов |

Эти файлы находятся в `.serena/memories/` и отслеживаются дашбордами.
