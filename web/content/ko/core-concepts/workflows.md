# 워크플로우

워크플로우는 oh-my-agent를 단순한 프롬프트 모음 이상으로 만드는 구조화된 시퀀스입니다. 슬래시 커맨드를 입력하면 정해진 단계, 품질 게이트, 완료 기준을 갖춘 다단계 프로세스가 시작됩니다.

## 전체 워크플로우 커맨드

### 대규모 오케스트레이션 워크플로우

**영속적(Persistent)** 워크플로우입니다 — 모든 태스크가 완료될 때까지 계속 실행되며, 실수로 중단되지 않습니다.

| 커맨드 | 하는 일 |
|---------|-------------|
| `/orchestrate` | CLI를 통해 에이전트를 병렬 스폰하고, 메모리로 조율하고, 진행 상황을 모니터링하고, 검증을 실행합니다. "모든 걸 해줘" 모드입니다. |
| `/coordinate` | 단계별 멀티 도메인 조율. PM이 먼저 기획하고, 에이전트가 크로스 리뷰와 QA 루프를 통해 실행합니다. |
| `/ultrawork` | 품질에 집착하는 워크플로우. 총 17단계 중 11개가 리뷰 단계인 5단계 프로세스. Plan → Implement → Verify → Refine → Ship. |

### 기획 및 탐색

| 커맨드 | 하는 일 |
|---------|-------------|
| `/plan` | PM 주도 태스크 분해. `.agents/plan.json`에 구조화된 계획을 출력합니다. |
| `/exec-plan` | `/plan`에서 만든 계획을 단계별로 실행합니다. |
| `/brainstorm` | 자유로운 아이디어 탐색. 구현에 착수하기 전 접근 방식을 탐색합니다. |
| `/deepinit` | 전체 프로젝트 초기화 — 코드베이스 분석, 컨벤션 설정, 도구 구성. |

### 품질 및 리뷰

| 커맨드 | 하는 일 |
|---------|-------------|
| `/review` | QA 리뷰: OWASP 보안, 성능, 접근성. qa-reviewer 에이전트에 위임합니다. |
| `/debug` | 구조화된 디버깅: 재현 → 진단 → 수정 → 회귀 테스트. |

### 디자인

| 커맨드 | 하는 일 |
|---------|-------------|
| `/design` | 7단계 디자인 워크플로우. 토큰, 컴포넌트 패턴, 접근성 규칙, 핸드오프 스펙이 담긴 DESIGN.md를 생성합니다. |

### 유틸리티

| 커맨드 | 하는 일 |
|---------|-------------|
| `/commit` | 변경 사항을 분석하고 적절한 type/scope의 Conventional Commit을 생성합니다. |
| `/setup` | 인터랙티브 프로젝트 구성. |
| `/tools` | MCP 서버 연결 관리. |
| `/stack-set` | 기술 스택 설정. |

## 슬래시 커맨드가 항상 필요한 건 아닙니다

oh-my-agent는 자연어에서 키워드를 감지하고 워크플로우를 자동 활성화합니다. "인증 기능을 기획해줘"라고 말하면 plan 워크플로우가 시작됩니다 — `/plan`을 입력할 필요 없이요.

**11개 언어**를 지원합니다 (영어, 한국어, 일본어, 중국어, 스페인어, 프랑스어, 독일어, 포르투갈어, 러시아어, 네덜란드어, 폴란드어).

"orchestrate가 뭐야?" 같은 질문은 정보 요청으로 인식되어 워크플로우가 실행되지 않습니다.

## 스킬 vs 워크플로우

간단히 구분하면:
- **스킬** = 에이전트의 전문성 (에이전트가 할 줄 아는 것)
- **워크플로우** = 오케스트레이션된 프로세스 (여러 에이전트가 함께 일하는 방법)

스킬은 "React 컴포넌트 만들기"일 수 있습니다. 워크플로우는 "기능 기획 → 컴포넌트 구현 → 보안 리뷰 → 코드 커밋"입니다.

## 일반적인 시퀀스

### 빠른 기능 개발
```
/plan → 결과 확인 → /exec-plan
```

### 복잡한 멀티 도메인 프로젝트
```
/coordinate → PM 기획 → 에이전트 스폰 → QA 리뷰 → 이슈 수정 → 출시
```

### 최고 품질 딜리버리
```
/ultrawork → 11개 리뷰 체크포인트가 있는 5단계 게이트 프로세스
```

### 버그 수정
```
/debug → 재현 → 근본 원인 → 수정 → 회귀 테스트
```
