# 대시보드 모니터링

여러 에이전트가 병렬로 실행 중일 때, 무슨 일이 벌어지고 있는지 보고 싶을 겁니다. 대시보드가 그 역할을 합니다.

## 대시보드 시작

```bash
# 터미널 UI
oma dashboard

# 웹 UI
oma dashboard:web
# → http://localhost:9847
```

## 추천 설정

터미널 3개를 나란히 사용하세요:

| 터미널 | 용도 |
|----------|---------|
| 1 | `oma dashboard` — 실시간 에이전트 상태 |
| 2 | 에이전트 스폰 명령 |
| 3 | 테스트 및 빌드 로그 |

팀 세션에서 공유 가시성을 위해 웹 대시보드를 브라우저에 열어두세요.

## 볼 수 있는 정보

대시보드는 `.serena/memories/`를 감시하며 다음을 보여줍니다:

- **세션 상태** — 실행 중, 완료, 또는 실패
- **태스크 보드** — 어떤 에이전트가 어떤 태스크를 맡고 있는지
- **에이전트 진행 상황** — 턴 수, 현재 활동
- **결과** — 도착하는 최종 산출물

업데이트는 이벤트 기반(파일 변경 감지)입니다 — CPU를 잡아먹는 폴링 루프가 없습니다.

## 문제 해결 신호

| 상황 | 대처 방법 |
|---------|-----------|
| "No agents detected" | 에이전트가 같은 `session-id`로 스폰되었는지 확인. `.serena/memories/`에 기록되고 있는지 확인. |
| 세션이 "running"에서 멈춤 | `progress-*` 파일 타임스탬프 확인. 더 명확한 프롬프트로 에이전트 재시작. |
| 웹에서 잦은 재연결 | 방화벽/프록시 확인. `dashboard:web` 재시작 후 페이지 새로고침. |
| 활동이 보이지 않음 | 오케스트레이터가 올바른 워크스페이스 디렉토리에 기록하고 있는지 확인. |

## 머지 전 확인

대시보드에서 빠르게 체크하세요:

- 모든 에이전트가 "completed" 상태에 도달
- 미해결 high-severity QA 발견 사항 없음
- 각 에이전트의 result 파일 존재
- 최종 산출물 이후 통합 테스트 실행

## 완료 시점

모니터링 단계는 다음 조건에서 완료됩니다:
- 세션이 터미널 상태(completed 또는 stopped)를 표시
- 활동 히스토리가 무슨 일이 있었는지 설명
- 완전한 가시성을 바탕으로 머지/릴리스 결정을 내림
