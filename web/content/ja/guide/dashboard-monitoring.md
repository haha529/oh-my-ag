# ダッシュボード監視

複数のエージェントを並列で実行している時、何が起きているか把握したいですよね。それがダッシュボードの役割です。

## ダッシュボードを起動する

```bash
# ターミナルUI
oma dashboard

# Web UI
oma dashboard:web
# → http://localhost:9847
```

## おすすめのセットアップ

3つのターミナルを並べて使いましょう：

| ターミナル | 用途 |
|----------|---------|
| 1 | `oma dashboard` — エージェントのライブステータス |
| 2 | エージェント起動コマンド |
| 3 | テストとビルドのログ |

チームセッション中は、共有で見えるようにWebダッシュボードをブラウザで開いておくのがおすすめです。

## 表示される情報

ダッシュボードは `.serena/memories/` を監視して以下を表示します：

- **セッションステータス** — 実行中、完了、または失敗
- **タスクボード** — どのエージェントがどのタスクを担当
- **エージェントの進捗** — ターン数、現在のアクティビティ
- **結果** — 到着した最終出力

更新はイベント駆動（ファイル変更検出）で、CPUを消費するポーリングループはありません。

## トラブルシューティングのシグナル

| 表示 | 対処法 |
|---------|-----------|
| "No agents detected" | 同じ `session-id` でエージェントが起動されているか確認。`.serena/memories/` に書き込みがあるか確認。 |
| セッションが "running" のまま | `progress-*` ファイルのタイムスタンプを確認。より明確なプロンプトでブロックされたエージェントを再起動。 |
| 頻繁な再接続（Web） | ファイアウォール/プロキシを確認。`dashboard:web` を再起動してページをリフレッシュ。 |
| アクティビティが表示されない | オーケストレーターが正しいワークスペースディレクトリに書き込んでいるか確認。 |

## マージ前に

ダッシュボードからのクイックチェックリスト：

- すべてのエージェントが "completed" ステータスに到達
- 未解決のhigh-severityなQA指摘がない
- 各エージェントの結果ファイルが存在
- 最終出力後にインテグレーションテストを実行

## 完了タイミング

監視フェーズは以下で完了です：
- セッションが終了状態を表示（completedまたはstopped）
- アクティビティ履歴が何が起きたかを説明している
- 完全な可視性を持ってマージ/リリースの判断ができた
