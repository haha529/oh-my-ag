# ワークフロー

ワークフローは、oh-my-agentを単なるプロンプト集以上のものにする構造化されたシーケンスです。スラッシュコマンドを入力すると、定義されたフェーズ、品質ゲート、完了条件を持つ多段階プロセスが開始されます。

## すべてのワークフローコマンド

### 大規模オーケストレーションワークフロー

これらは**永続的**です。すべてのタスクが完了するまで動き続け、誤って中断されることはありません。

| コマンド | 機能 |
|---------|-------------|
| `/orchestrate` | CLIを通じてエージェントを並列起動し、メモリで連携、進捗を監視、検証を実行。「全部やる」モード。 |
| `/coordinate` | ステップバイステップのマルチドメイン連携。PMが先に計画し、エージェントがクロスレビューとQAループで実行。 |
| `/ultrawork` | 品質にこだわるワークフロー。5フェーズ、全17ステップ中11がレビュー。Plan → Implement → Verify → Refine → Ship。 |

### 計画と探索

| コマンド | 機能 |
|---------|-------------|
| `/plan` | PM主導のタスク分解。`.agents/plan.json` に構造化されたプランを出力。 |
| `/exec-plan` | `/plan` で作成したプランをステップバイステップで実行。 |
| `/brainstorm` | 自由なアイデア出し。実装に着手する前にアプローチを探索。 |
| `/deepinit` | フルプロジェクト初期化 — コードベース分析、規約設定、ツール構成。 |

### 品質とレビュー

| コマンド | 機能 |
|---------|-------------|
| `/review` | QAレビュー：OWASPセキュリティ、パフォーマンス、アクセシビリティ。qa-reviewerエージェントに委任。 |
| `/debug` | 構造化デバッグ：再現 → 診断 → 修正 → リグレッションテスト。 |

### デザイン

| コマンド | 機能 |
|---------|-------------|
| `/design` | 7フェーズのデザインワークフロー。トークン、コンポーネントパターン、アクセシビリティルール、ハンドオフ仕様を含むDESIGN.mdを作成。 |

### ユーティリティ

| コマンド | 機能 |
|---------|-------------|
| `/commit` | 変更を分析し、適切なtype/scopeでConventional Commitを作成。 |
| `/setup` | 対話型プロジェクト設定。 |
| `/tools` | MCPサーバー接続の管理。 |
| `/stack-set` | 技術スタックの設定。 |

## スラッシュコマンドは必須ではない

oh-my-agentは自然言語のキーワードを検出して、ワークフローを自動起動します。「認証機能を計画して」と言えば、`/plan` なしでplanワークフローが開始されます。

これは**11言語**（英語、韓国語、日本語、中国語、スペイン語、フランス語、ドイツ語、ポルトガル語、ロシア語、オランダ語、ポーランド語）で動作します。

「orchestrateとは何？」のような質問は情報要求として認識され、何もトリガーしません。

## スキルとワークフローの違い

シンプルな区別です：
- **スキル** = エージェントの専門知識（エージェントが何をできるか）
- **ワークフロー** = オーケストレーションされたプロセス（複数のエージェントがどう連携するか）

スキルは「Reactコンポーネントを作る」。ワークフローは「機能を計画 → コンポーネントを構築 → セキュリティをレビュー → コードをコミット」。

## 典型的なシーケンス

### クイック機能追加
```
/plan → 出力を確認 → /exec-plan
```

### 複雑なマルチドメインプロジェクト
```
/coordinate → PMが計画 → エージェント起動 → QAレビュー → 問題修正 → リリース
```

### 最高品質デリバリー
```
/ultrawork → 11のレビューチェックポイントを含む5フェーズゲートプロセス
```

### バグ修正
```
/debug → 再現 → 根本原因 → 修正 → リグレッションテスト
```
