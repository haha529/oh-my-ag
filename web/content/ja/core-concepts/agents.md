# エージェント

oh-my-agentのエージェントは汎用チャットボットではありません。それぞれが特定のエンジニアリングチームの役割をモデル化しており、スコープ、ツール、品質基準が明確に定義されています。

## エージェントカテゴリ

エンジニアリング組織のように考えてみてください：

| カテゴリ | エージェント | 担当領域 |
|----------|--------|-----------------|
| **アイデア出し** | oma-brainstorm | コードを書く前にアイデアを探索 |
| **計画** | oma-pm | 要件定義、タスク分解、APIコントラクト |
| **実装** | oma-frontend, oma-backend, oma-mobile, oma-db | 実際のコーディング |
| **デザイン** | oma-design | デザインシステム、トークン、UI/UXパターン |
| **インフラ** | oma-tf-infra | Terraformによるクラウドプロビジョニング |
| **DevOps** | oma-dev-workflow | CI/CD、マイグレーション、リリース |
| **品質** | oma-qa, oma-debug | レビュー、セキュリティ監査、バグ修正 |
| **ローカライゼーション** | oma-translator | 自然な多言語翻訳 |
| **コーディネーション** | oma-orchestrator, oma-coordination | エージェントの実行と同期 |
| **Git** | oma-commit | クリーンなConventional Commits |

## エージェントの実行プロセス

すべてのエージェントが同じ規律に従います：

### 1. チャータープリフライト

コードを書く前に、エージェントは `CHARTER_CHECK` を宣言します：
- どのドメインで作業するか
- どの制約が適用されるか
- どのような前提を置いているか
- 「完了」とは何か

これにより、スコープのずれや誤解を早期に検出できます。

### 2. 二層ローディング

エージェントはトークン効率を重視した設計です：
- **レイヤー1**（`SKILL.md`、約800バイト）：アイデンティティとルーティングルール — 常にロード
- **レイヤー2**（`resources/`）：実行プロトコル、エラープレイブック、コードテンプレート — 必要な時だけロード

すべてを事前にロードする場合と比較して、約75%のトークンを節約します。

### 3. スコープ限定実行

フロントエンドエージェントはバックエンドコードに触れません。DBエージェントはUIコンポーネントを変更しません。各エージェントは自分の担当領域を守ります。

### 4. 品質ゲート

すべてのエージェントにドメイン固有のチェックリストがあります。チェックリストをパスするまで作業は「完了」になりません。

## ワークスペース戦略

マルチエージェントプロジェクトでは、ワークスペースを分けることでマージコンフリクトを減らします：

```text
./apps/api      → backendエージェントのワークスペース
./apps/web      → frontendエージェントのワークスペース
./apps/mobile   → mobileエージェントのワークスペース
```

## オーケストレーションフロー

マルチエージェントワークフローを実行すると：

1. **PMエージェント**がタスクをドメイン固有のサブタスクに分解
2. **ドメインエージェント**がそれぞれのワークスペースで並列実行
3. **進捗**がSerenaメモリ（`.serena/memories/`）にストリーミング
4. **QAエージェント**がクロスドメインの整合性を検証
5. **結果**が収集され、マージ準備完了

## ランタイム状態（Serenaメモリ）

エージェントは共有メモリファイルを通じて連携します：

| ファイル | 目的 |
|------|---------|
| `orchestrator-session.md` | 現在のセッション状態 |
| `task-board.md` | タスクの割り当てとステータス |
| `progress-{agent}.md` | エージェントごとの進捗更新 |
| `result-{agent}.md` | エージェントの最終出力 |

これらは `.serena/memories/` に配置され、ダッシュボードが監視する対象です。
